# User Story: Generate a Landing Zone for a VM in Azure

## Story

**As a** cloud infrastructure engineer,
**I want to** provision a standardized landing zone for a virtual machine in Azure,
**So that** the VM is deployed in a secure, well-governed, and network-ready environment that follows organizational best practices.

---

## Background

A landing zone is a pre-configured Azure environment (resource group, virtual network, subnets, NSGs, IAM roles, and policies) that provides a consistent and compliant foundation before deploying any workload VM.

---

## Acceptance Criteria

- [ ] A dedicated **Resource Group** is created with appropriate tags (environment, owner, cost-center).
- [ ] A **Virtual Network (VNet)** with at least one subnet is provisioned within the resource group.
- [ ] A **Network Security Group (NSG)** is attached to the subnet with baseline deny-all inbound rules and allow rules for required ports (e.g., SSH/RDP restricted to trusted IPs).
- [ ] **IAM role assignments** are configured so only authorized principals have access (Contributor for ops team, Reader for auditors).
- [ ] **Azure Policy** assignments enforce tagging, allowed VM SKUs, and allowed regions.
- [ ] A **diagnostic settings** configuration routes activity logs to a Log Analytics Workspace.
- [ ] The landing zone is provisioned via **Infrastructure-as-Code** (Bicep or Terraform) and stored in version control.
- [ ] The deployment is **idempotent** — re-running the IaC does not cause errors or duplicate resources.
- [ ] A **validation step** confirms all resources are in a succeeded provisioning state after deployment.

---

## Subtasks

1. **Define landing zone scope** — agree on region, resource naming convention, and tagging taxonomy.
2. **Create IaC template** — write Bicep/Terraform modules for resource group, VNet, subnet, and NSG.
3. **Configure NSG rules** — define baseline inbound/outbound rules and document allowed ports.
4. **Set up IAM assignments** — assign built-in roles to the relevant Azure AD groups/service principals.
5. **Apply Azure Policy** — assign policy initiatives for tagging compliance and allowed resource types.
6. **Wire up diagnostics** — configure diagnostic settings to forward logs to Log Analytics Workspace.
7. **CI/CD pipeline** — add a pipeline step (GitHub Actions / Azure DevOps) to lint, validate, and deploy the IaC.
8. **Write smoke tests** — verify resource existence, NSG rule correctness, and RBAC assignments post-deploy.
9. **Document the landing zone** — create a runbook describing the architecture, access model, and how to extend it.

---

## Definition of Done

- All acceptance criteria are checked off.
- IaC code is peer-reviewed and merged to the main branch.
- Deployment has been validated in the target Azure subscription.
- Documentation is updated and accessible to the team.
